# هل نبني على اللعبة القديمة أم نبدأ لعبة جديدة؟

## القرار التنفيذي

ابدأ لعبة جديدة، مع الاستفادة الانتقائية من اللعبة القديمة فقط إذا كانت تحتوي على منطق لعب مفيد أو بيانات جاهزة.

## السبب

الهوية الجديدة ليست مجرد تغيير ألوان. هي إعادة بناء كاملة لتجربة المنتج:

- نظام شاشات جديد.
- نظام ملاحة جديد.
- هوية بصرية مختلفة.
- أسلوب بطاقات ومهام ومتجر وتقدم.
- واجهة لعب HUD جديدة.
- تجربة مناسبة للأطفال 7–14.
- قابلية PWA/جوال.

بناء هذه الهوية فوق كود قديم غالبًا سيؤدي إلى ترقيع، تعارضات CSS/JS، صعوبة صيانة، وبطء في التطوير.

## متى نستفيد من القديم؟

استخرج من اللعبة القديمة فقط:

1. أي خوارزمية حركة جيدة.
2. أي منطق احتساب نتيجة أو نقاط.
3. أي بيانات لاعبين أو فرق إذا كانت منظمة.
4. أي مؤثرات صوتية أو رسومية أصلية ومملوكة لك.
5. أي Backend جاهز صالح.

## خطة الدمج الصحيحة

- اجعل هذا المشروع هو الأساس الجديد.
- أنشئ مجلدًا مؤقتًا باسم `legacy_review`.
- افحص ملفات اللعبة القديمة.
- انقل المنطق المفيد فقط بعد إعادة كتابته وفق بنية هذا المشروع.
- لا تنقل CSS أو UI قديم إلا إذا كان مطابقًا للهوية الجديدة.

## القرار العملي

```text
New Build = نعم
Reuse Legacy Logic = انتقائي فقط
Reuse Legacy UI = لا
Reuse Legacy Assets = فقط إذا كانت أصلية ومملوكة
```
