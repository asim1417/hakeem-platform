## 2) 🔴 الثغرة الحرجة (موثّقة — تتطلب قرار تشغيلي)
**`DISABLE_AUTH=true` افتراضيًا** → في هذا الوضع يُنشأ «زائر النظام» بدور **SYSTEM_ADMIN** كامل الصلاحيات، فيُفتح الموقع كاملاً (الإدارة، المستخدمون، التدقيق) للعامة.

**لماذا لم يُعكس آليًا في هذه المرحلة:** عكسه يكسر تجربة العرض الحالية التي تعتمد الوصول بلا تسجيل، وهو **قرار تشغيلي** يخصّ بيئة النشر لا الكود.

**التصليب للإنتاج (خطوة واحدة):**
```bash
DISABLE_AUTH=false          # تفعيل تسجيل الدخول الإلزامي
AUTH_SECRET=<سرّ-قوي-عشوائي>  # استبدال السرّ الافتراضي
```
بعدها: middleware يعيد التوجيه إلى `/login`، وتُطبّق الأدوار الفعلية. المستخدم الإداري يُنشأ بالبذر (`admin@hakeem.local`).

## 3) RBAC — الأدوار والصلاحيات
- **الأدوار (4):** `SYSTEM_ADMIN` · `LAWYER` · `TRAINER` · `TRAINEE`.
- **الصلاحيات (14):** استشارات/محاكاة/تدريب/مكتبة/نواة(عرض/تحرير/إدارة)/مرفقات/مستخدمون/تقارير/تدقيق.
- مصدر مزدوج: ثابت في `rbac.ts` + جداول DB (`RoleRecord/RolePermission`) مع دمج ذكي. `SYSTEM_ADMIN` يتجاوز كل الفحوص.
- **مطابقة الأدوار المقترحة (admin/lawyer/researcher/user):** `SYSTEM_ADMIN→admin`, `LAWYER→lawyer`, `TRAINER/TRAINEE→researcher/user`. (إعادة التسمية اختيارية، تتطلب migration بيانات — مؤجّلة لتفادي الكسر.)
