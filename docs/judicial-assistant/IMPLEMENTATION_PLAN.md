# خطة التنفيذ — المعاون القضائي السعودي

خدمةٌ مستقلّة داخل حكيم، منفصلة عن «اسأل حكيم» وأوضاعه الستّة. تُبنى على الأساس القائم لا بجانبه.

## مبدأ الفصل
- المسار: `/dashboard/judicial-assistant/*` (لا يمسّ `/dashboard/ask`).
- الوحدة البرمجيّة: `lib/modules/judicial-assistant/*` + `components/judicial-assistant/*`.
- إعادة الاستخدام لا التكرار: التصميم (`components/ui/legal`, tokens)، التدقيق (`auditEvent`)،
  الاسترجاع المؤصَّل (`createAgentConsultationDraft`)، الصلاحيات (`requirePagePermission`).

## ما نُفِّذ في هذه المرحلة (Sprint 1 — §72)
- بنية النطاق: `types.ts`, `catalog.ts` (24 خدمة + 12 مرحلة + خريطة اقتراح), `labels.ts`.
- موصل: `CaseManagementConnector` + `MockCaseConnector` (قضيّتان صناعيّتان، قراءة فقط).
- طبقة الوصول: `store.ts` (لوحة القاضي + قوائم + قضية).
- JS-001 حيّ: `summary.ts` عبر الوكيل المؤصَّل (حجب صادق، لا اختلاق، استشهاد بمعرّفات حقيقيّة).
- API: `POST /api/judicial-assistant/summary` (RBAC + تدقيق).
- الواجهات: لوحة القاضي، قائمة القضايا، مساحة القضية (رأس + شريط مراحل + خريطة + أعمال مقترحة).
- الأيقونات (§24)، التنقّل (عنصر «المعاون القضائيّ» بأيقونة Scale)، CSS الوحدة، i18n.
- الوثائق: هذا الملفّ + GAP_ANALYSIS + ADR-000.

## قواعد منع الانحراف (§78) — ملتزَمة
- ليست دردشة: مساحة قضيةٍ تقودها المرحلة.
- منسّق واحد لا 20 وكيلًا حرًّا: الأعمال تُقترح مركزيًّا بحسب المرحلة.
- قاعدة البيانات مصدر الحقيقة (لاحقًا) لا Vector DB.
- لا نصّ نظاميّ من الذاكرة: كلّ استشهادٍ مؤصَّل أو حجبٌ صادق.
- لا Auto-Approve: كلّ مخرَجٍ مسودّة تحتاج تثبيتًا بشريًّا.
- لا كتابة خارجيّة، ولا بيانات حقيقيّة، ولا تكامل وهميّ مع «تقاضي».

## معايير القبول لهذه المرحلة
- يفتح `/dashboard/judicial-assistant` ويعرض أولويّات اليوم من الموصل الصناعيّ.
- تفتح القضية وتعرض الرأس والمرحلة والخريطة والأعمال المقترحة بحسب مرحلتها.
- يعمل JS-001 مؤصَّلًا: استشهادات قابلة للفتح أو حجبٌ صادق، مع سجلّ تدقيق.
- RTL كامل، بلا بيانات حقيقيّة، بلا مسٍّ بالمصادقة أو «اسأل حكيم».
