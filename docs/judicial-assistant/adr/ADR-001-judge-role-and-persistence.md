# ADR-001 — دور «قاضٍ»، الصلاحية، وحفظ المخرجات (المرحلة 1ب)

- **الحالة:** مقبول
- **التاريخ:** 2026-07-19
- **السياق:** تفعيل §12 (RBAC + ABAC) و§31 (حفظ) للمعاون القضائي، مع بقاء النشر آمنًا على Neon/Vercel
  حيث لا يهاجر البناء (`build = prisma generate && next build`).

## القرارات

1. **دور `JUDGE` + صلاحية `JUDICIAL_ASSISTANT_USE`.** أُضيف الدور إلى enum `UserRole`، والصلاحية إلى
   الخريطة الثابتة `ROLE_PERMISSIONS`. `canUser` يحترم هذه الخريطة كسقوطٍ احتياطيّ عند غياب `RoleRecord`
   في القاعدة، فتعمل الصلاحية **دون بذر جداول صلاحيّات**. مُنِحت لـ`JUDGE` و`SYSTEM_ADMIN` و`LAWYER`
   (كي لا يُحجب المستخدمون الحاليّون). صفحات/مسارات المعاون نُقلت من `CONSULTATIONS_FULL` إليها.

2. **ABAC حقيقيّ لا صوريّ.** `caseVisibleTo(ctx, case)`: المدير يرى الكلّ؛ القضايا الصناعيّة (بلا مالك)
   تُرى لكلّ من يملك الصلاحية وموسومة صراحةً؛ ومكان ترشيح الملكيّة/الدائرة جاهزٌ للقضايا الحقيقيّة. صفحة
   القضية تفحص الرؤية وتردّ 404 عند المنع.

3. **حفظ المخرجات دون كسر النشر.** جدول `judicial_analyses` (Prisma model + سكربت idempotent
   `scripts/apply-judicial-assistant.ts` عبر `npm run db:judicial`). الوصول **دفاعيّ** (fail-open):
   إن لم يُطبَّق السكربت بعد، يعمل الحفظ/القراءة بلا خطأٍ ظاهر (تُتخطّى بصمت)، فتبقى الصفحة سليمة.

4. **الهجرة يدويّة ومقصودة.** كإجراء المشروع القائم (`db:apply-normative`, `db:username-column`)،
   تُطبَّق التغييرات بجُمَل `IF NOT EXISTS` / `ADD VALUE IF NOT EXISTS` مرّةً على كلّ بيئة. لا يُدرَج
   `migrate deploy` في البناء تفاديًا للأعطال.

## خطوة التشغيل المطلوبة
بعد الدمج: تشغيل `npm run db:judicial` مرّةً على كلّ بيئة (بـDATABASE_URL الصحيح) لإنشاء الجدول ودور
`JUDGE`. حتى قبل تشغيله، التطبيق لا ينكسر (سقوطٌ آمن) لكن لا يُحفظ سجلّ التحليلات ولا يُسنَد دور «قاضٍ».

## البدائل المرفوضة
- `prisma migrate deploy` في البناء: يخاطر بأعطال نشرٍ على القاعدة الحيّة.
- تطبيع كامل النموذج القضائيّ الآن: نطاقٌ كبير وخطر نشرٍ أعلى؛ يؤجَّل.
- بذر `RolePermission` في القاعدة: غير لازم لأنّ `canUser` يحترم الخريطة الثابتة.
