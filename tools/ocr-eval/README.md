# أداة قياس دقة الاستخراج النصي — `ocr_eval`

أداة تقيس وتقارن محرّكات الاستخراج النصي (OCR) على مجموعة عربية قانونية موسومة،
وتُخرج ترتيباً موضوعياً بأقلّ خطأ. هدفها سدّ الفجوة الأخطر في المنظومة: **غياب قياس
الدقة**. لا تُحسّن الأداة الاستخراج بذاتها؛ بل تقيسه لتوجيه قرار المحرّك والتحسين.

---

## لماذا هذه الأداة؟

القرار الاستراتيجي هو **تنسيق أفضل المحرّكات لا منافستها على مستوى البكسل**. ولاتخاذ
هذا القرار بأدلّة، نحتاج أن نقيس — على وثائقنا نحن — أيّ محرّك أدقّ ولأيّ نوع صفحة،
وكم يكلّف. هذه الأداة توفّر هذا القياس المتكرّر.

---

## بنية مجموعة القياس

كل عيّنة ثلاثة ملفات في مجلد المجموعة `goldset/<name>/`:

```
<id>.png        صورة الصفحة (أو .jpg / .tif)
<id>.gt.txt     الحقيقة الأرضية: النص الصحيح، مُراجَع بشرياً حرفاً بحرف
<id>.meta.json  اختياري: نوع الوثيقة + الكيانات الصحيحة (لقياس الاستخراج لاحقاً)
```

مثال `meta.json`:
```json
{"doc_type":"حكم / صك",
 "entities":{"parties":["شركة الأفق التجارية"],"amounts":["331000000"],"deed_numbers":["31420786"]}}
```

مجلد `goldset/sample/` يحوي عيّنة اصطناعية جاهزة لتجربة الأداة فوراً.

> ملاحظة: صورة العيّنة الاصطناعية مولّدة بخط غير قانوني، فحصول Tesseract على درجة
> منخفضة عليها **متوقّع** ولا يعكس أداءه الحقيقي؛ المجموعة الفعلية صفحات مسح حقيقية.

---

## بروتوكول بناء المجموعة الذهبية (مقترَح)

الجودة تأتي من مجموعة تمثيلية مُتقنة الوسم، لا من حجمها. المقترَح:

1. **الحجم:** ٥٠–١٠٠ صفحة كبداية معتبرة إحصائياً؛ تُوسَّع تدريجياً.
2. **التمثيل:** وزّع الصفحات على أنواع الوثائق الفعلية (أحكام، عقود، تقارير خبرة،
   محاضر، مراسلات) وعلى درجات جودة المسح (نظيف / متوسط / رديء) بما يشبه الواقع.
3. **الوسم:** تُفرّغ الحقيقة الأرضية يدوياً بدقّة تامة، مع مراجعة ثانية مستقلة لعيّنة
   منها لضبط اتساق الوسم.
4. **الأرقام والأسماء:** تُنقل حرفياً كما في الأصل (لا تُصحَّح إملائياً)، لأن دقّة
   الأرقام والأسماء هي جوهر القيمة القانونية.
5. **الخصوصية:** إن استُخدمت وثائق قضايا حقيقية، تُعالَج محلياً فقط، ويُفضّل تعمية
   الأسماء الحسّاسة في نسخة القياس المتداولة.

---

## التشغيل

```bash
# تجربة فورية (محرّك محاكاة، بلا تبعيات):
python run_eval.py --gold goldset/sample --engines mock

# مقارنة محرّكات حقيقية على مجموعتك:
python run_eval.py --gold goldset/legal_v1 --engines tesseract azure_di google_docai
```

المحرّكات السحابية تُفعَّل بضبط مفاتيحها في البيئة، وتبقى معطّلة تلقائياً إن غابت:

```
AZURE_DI_ENDPOINT=...   AZURE_DI_KEY=...
GCP_DOCAI_PROCESSOR=projects/.../processors/...   GOOGLE_APPLICATION_CREDENTIALS=...
```

المخرجات في `reports/`: ملخّص Markdown (ترتيب المحرّكات) + CSV تفصيلي لكل عيّنة.

---

## تفسير المقاييس

- **CER** معدّل خطأ الأحرف، و**WER** معدّل خطأ الكلمات — كلّما قلّ كان أفضل (0 = مطابقة تامة).
- **الدقة %** = (1 − CER) × 100.
- **خام مقابل مُطبَّع:** النسخة «الخام» تحاسب على التشكيل واختلاف الهمزات والتاء المربوطة؛
  والنسخة «المُطبَّعة» تتجاهل ما لا يؤثر في الاسترجاع القانوني. الفرق بينهما يكشف كم من
  «الخطأ» هو إملائي غير مؤثّر، وكم هو خطأ حقيقي في المحتوى.
- **الكيانات (لاحقاً):** `metrics.entity_prf` يقيس دقة/استرجاع/F1 على الأطراف والمبالغ
  والصكوك، فيربط القياس بالقيمة القانونية لا بالحروف فقط.

### عتبات قبول مقترَحة (تُضبط بالتجربة)

| المستوى | CER مُطبَّع | الدلالة |
|---|---|---|
| ممتاز | أقل من 0.02 | جاهز للاعتماد بمراجعة خفيفة |
| مقبول | 0.02 – 0.08 | صالح للبحث، يحتاج تدقيق المواضع الحرجة |
| ضعيف | أكثر من 0.08 | يُعاد استخراجه بمحرّك أقوى أو يُحال للمراجعة |

## الملفات

```
ocr_eval/
  metrics.py     مقاييس CER/WER + التطبيع العربي + قياس الكيانات
  engines.py     واجهة المحرّكات (Mock / Tesseract / Azure / Google)
  run_eval.py    مشغّل القياس ومولّد التقارير
  goldset/       مجموعات القياس (sample جاهزة)
  reports/       التقارير المولّدة
```

التبعيات: `Pillow` (لتوليد عيّنات فقط)، و`pytesseract`/`tesseract-ara` أو مفاتيح
سحابية بحسب المحرّكات المختارة. المقاييس نفسها بلا تبعيات خارجية.

## المحرّكات المفتوحة عالية الجودة (محلية)

أُضيفت محوّلات حقيقية بنمط lazy-import، تُكتشف تلقائياً عند التثبيت. لعرض حالتها:

```bash
python engines.py            # جدول: المحرّك | متوفّر | يسترجع جداول؟
```

| المحرّك | يسترجع جداول | الرخصة | ملاحظة |
|---|---|---|---|
| easyocr | لا | Apache-2.0 | نص عربي/إنجليزي محلي |
| paddleocr | لا | Apache-2.0 | نص متعدّد اللغات |
| paddleocr_structure | نعم | Apache-2.0 | PP-StructureV3 — للقوائم المالية |
| surya | نعم | GPL-3.0 | 90+ لغة؛ انتبه للترخيص التجاري |
| docling | نعم | MIT | جداول TableFormer؛ مناسب للبيئات المعزولة |

التثبيت الانتقائي في `requirements-ocr.txt`. مثال لقياس محرّكات الجداول على قائمة مالية:

```bash
pip install docling paddleocr paddlepaddle
python run_eval.py --gold goldset/financial_v1 --engines docling paddleocr_structure
```

> صريح: هذه المحوّلات كُتبت بنيوياً وتتحقق من التوفّر بلا انهيار، لكنها لم تُشغَّل
> end-to-end في بيئة الإعداد (النماذج ثقيلة). تُفعَّل بالتثبيت على جهازك.

## مكتبات القياس

المقاييس تستخدم `rapidfuzz` (مسافة التحرير) و`jiwer` (WER) عند توفّرهما، مع تراجع
آمن إلى تنفيذ بايثون خام إن غابتا. التطبيع العربي من إعدادنا (لا مكتبة عامة تؤدّيه).

---

## التكامل مع منصّة الوثائق (حكيم)

- **محرّك `tesseract_prep`**: يطبّق **المعالجة المسبقة نفسها** المستخدمة في المنصّة
  (تدرّج رمادي + عتبة Otsu + تكبير الصور الصغيرة إلى ~300DPI) قبل Tesseract.
  مقارنته بـ `tesseract` الخام تُظهر مكسب الجودة رقمياً على وثائقك.
- **مجموعة `goldset/legal`**: صفحات قانونية عربية مولّدة بنصّ حقيقة معروف —
  نظيفة + متدهورة (ضوضاء/تباين منخفض) لقياس أثر المعالجة على المسح الرديء.
- **workflow**: `OCR Accuracy Eval` (Actions → Run) يثبّت Tesseract العربي
  ويشغّل المقارنة `mock,tesseract,tesseract_prep` ويرفع التقارير (CER/WER).

> القياس على وثائق حقيقية: أضف صفحاتك المسوّمة في `goldset/<name>/` وشغّل الـ workflow
> بمُدخل `goldset=goldset/<name>`. الأرقام والأسماء تُنقل حرفياً كما في الأصل.
